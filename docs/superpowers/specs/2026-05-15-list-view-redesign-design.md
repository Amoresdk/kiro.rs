# 凭据管理面板列表视图改造

- **状态**: 待实施
- **撰写日期**: 2026-05-15
- **作者**: Claude（基于与逆向号池管理员的需求对齐）
- **影响范围**: 仅前端 `admin-ui/`，不改后端 API
- **预计工作量**: 中等（~10 个新组件 + 1 个组件删除 + 数据派生层重写）

## 1. 背景与目标

### 1.1 现状

`admin-ui` 当前的凭据管理界面采用**卡片网格视图**（`src/components/credential-card.tsx`），每页固定 12 张卡，3 列网格布局。每张卡呈现单个凭据的全部信息（邮箱、优先级、失败次数、订阅、用量、最后调用、6 个操作按钮）。

### 1.2 痛点（号池管理员视角）

| 问题 | 描述 |
|---|---|
| 信息密度低 | 1080P 屏幕一屏只能看到 6 张卡，号池超过 50 个时翻页和横向跳页都很累 |
| 横向比较困难 | 比较两个凭据的剩余用量、优先级时眼睛要在卡片间跳动 |
| 缺少筛选/排序 | 只能按默认顺序看，无法按"剩余最少 / 失败最多 / 最久未调用"快速排序 |
| 批量瞄准难 | 勾选 10 个号需要在 12 张卡中精确点击复选框 |
| 操作过密 | 每张卡 6 个按钮，单条操作过于密集 |
| 分页粒度小 | 12/页对"号池盘点"工作流太碎 |

### 1.3 目标

把凭据管理面板从「凭据展示页」升级为「**号池作业台**」：
- 单屏可扫视的凭据数量提升 4–8 倍
- 支持按状态、订阅、认证方式、关键字筛选
- 支持按用量、失败次数、最后调用、优先级排序
- 批量操作更高效（更窄的复选框列、更紧凑的行高）
- 关键状态（异常/限速/超额）通过颜色一眼可识别

### 1.4 非目标

- 不修改后端 `/api/admin/credentials` 接口
- 不引入 react-router、redux 等新基础设施
- 不支持移动端深度优化（< 768px 显示桌面提示）
- 不保留卡片视图作为可切换选项（直接替换）

## 5. 交互行为

### 5.1 筛选 / 排序 / 搜索的状态同步

- 所有筛选、排序、搜索、分页状态写入 URL query：`?status=&sub=&auth=&q=&sort=&dir=&page=&size=`
- 浏览器刷新后状态保留
- URL 可分享给同事

### 5.2 复合筛选语义

- 状态 tab、订阅、认证、搜索关键字之间是 **AND** 关系
- 认证多选内部是 **OR** 关系（选中 Social + IdC = 命中其中之一）

### 5.3 搜索匹配规则

输入字符串 `q`（去除首尾空白后）命中以下任一即视为命中（不区分大小写）：

- `cred.email` 包含 `q`（子串匹配）
- `cred.maskedApiKey` 的末尾 4 个字符 === `q`（仅当 `q.length === 4` 且 `q` 全部为字母或数字）。`maskedApiKey` 形如 `sk-***...3a7f`，取字符串末尾 4 字符做对比
- `cred.id.toString() === q`（精确匹配）

### 5.4 排序

- 4 列可排序：用量、优先级、失败、最后调用
- 默认排序：按 `id` 升序（即"全部不排序"时的稳定顺序）
- 排序键空值处理：用量未查询的行排在最末（升降序均如此）；最后调用为 null 的行排在最末

### 5.5 行展开

- 点击行首 ▸ / ▾ 切换展开
- 多行可同时展开
- 展开状态不写入 URL，刷新即重置
- 展开行内"立即查询"按钮触发该凭据的 balance 查询

### 5.6 选中

- 点击复选框切换选中
- 表头复选框：当前页全选 / 取消全选
- 切换页 / 切换筛选 不清空已选中（但筛选后不可见的选中项不参与批量操作）
- 批量操作完成后清空选中

### 5.7 行内点击优先级编辑

沿用当前卡片视图：点击优先级数字 → 切换为输入框 + ✓ ✕ 按钮 → ✓ 提交 / ✕ 取消。

### 5.8 移动端守卫

`window.innerWidth < 768px` 时整页替换为：

```
┌─────────────────────────┐
│  请在桌面端访问 Kiro    │
│  Admin                  │
│                         │
│  当前界面针对桌面端优化 │
│  移动端体验受限         │
└─────────────────────────┘
```

### 5.9 暗色模式

沿用当前 dark mode 切换；所有色值在 dark 模式下走 Tailwind 的 dark: 变体。

## 6. 数据派生

### 6.1 状态派生

```
deriveStatus(cred):
  if cred.disabled || cred.refreshFailureCount >= 1:
    return 'error'
  if cred.failureCount > 0:
    return 'throttled'
  return 'normal'
```

### 6.2 用量段派生

```
deriveUsageSegment(balance):
  if balance == null:                       return 'unknown'
  if balance.usageLimit <= 0:               return 'unknown'   // 防除零
  pct = balance.currentUsage / balance.usageLimit
  if pct < 0.8:                             return 'normal'
  if pct < 1.0:                             return 'warning'
  if pct === 1.0:                           return 'full'
  return 'overflow'
```

### 6.3 派生位置

派生函数放在 `src/lib/derive.ts`，纯函数无副作用。CredentialRow 内部按需调用。

### 6.4 缓存

派生结果不做 memoization（数据量百级，纯计算成本可忽略）。如果未来数据量飙升再加 useMemo。

## 8. URL 状态管理

### 8.1 URL 参数

| 参数 | 含义 | 默认值 | 取值 |
|---|---|---|---|
| `status` | 状态 tab | `all` | `all` / `normal` / `throttled` / `error` |
| `sub` | 订阅筛选 | 空 | 字符串（订阅标题） |
| `auth` | 认证筛选 | 空 | 逗号分隔：`social,idc,api_key` |
| `q` | 搜索关键字 | 空 | 字符串 |
| `sort` | 排序列 | 空 | `usage` / `priority` / `failure` / `lastUsed` |
| `dir` | 排序方向 | `desc` | `asc` / `desc` |
| `page` | 页码 | `1` | 正整数 |
| `size` | 每页条数 | `50` | `50` / `100` / `200` |

### 8.2 实现方式

不引入 react-router。用 `URLSearchParams` + 自定义 `useUrlState` hook（位于 `src/hooks/use-url-state.ts`），基于 `history.replaceState` 同步。

## 9. 字段补全（依赖后端确认）

主表「最后调用」列依赖 `lastUsedAt`，已存在；「成功」列依赖 `successCount`，已存在；「失败」列依赖 `failureCount` + `refreshFailureCount`，均已存在。**无需后端改动**。

`balance` 信息仍按当前模式（每行独立查询，由前端 `balanceMap` 缓存）。"查询信息" 按钮逐个查询当前页凭据的 balance；展开行的"立即查询"按钮单独查询单个凭据。

## 10. 性能与边界

### 10.1 数据量

号池规模假设：< 500 个凭据。在此规模下：
- 全量前端筛选/排序/搜索性能可忽略
- 不需要虚拟滚动
- 不需要后端分页

如果未来规模超 1000，需考虑切换为后端分页 + 服务端筛选。本次改造不预留此能力。

### 10.2 边界情况

| 边界 | 处理 |
|---|---|
| 凭据列表为空 | 表体显示「暂无凭据 [+ 添加凭据]」空态卡 |
| 筛选后无结果 | 表体显示「没有匹配的凭据 [清空筛选]」空态卡 |
| 当前页超出筛选后总页数 | 自动跳到最后一页 |
| balance.usageLimit = 0 | 进度条显示为「未查询」灰空槽，避免除零 |
| email 字段缺失 | 邮箱列降级显示 `凭据 #{id}`，搜索仍可用 ID/maskedApiKey 命中 |
| 非常长的邮箱 | 邮箱列 CSS `text-overflow: ellipsis`，悬停显示完整邮箱 tooltip |

## 11. 可访问性

- 所有可点元素提供 `aria-label`（图标按钮、排序表头、展开图标）
- 状态徽章使用文字 + 颜色双重编码，色弱用户可识别
- 复选框使用原生 input 或 `@radix-ui/react-checkbox`，键盘可达
- 表格使用语义 `<table>` 而非 `<div>` grid，屏幕阅读器友好
- 排序表头同时有 ARIA `aria-sort` 属性

## 13. 风险与折中

| 风险 | 描述 | 缓解 |
|---|---|---|
| 删除卡片视图后用户不适应 | 老用户依赖卡片视图的视觉习惯 | git 历史保留可回滚；改造前在群里通知；首屏空态可提示新视图特性 |
| 状态判定与实际场景偏差 | 当前规则下可能将"瞬时网络抖动导致的 failureCount=1"误判为限速 | 在 spec 中暴露规则；如运行后发现误报多，调阈值即可（修改一行代码） |
| URL 参数膨胀 | 8 个参数让 URL 较长 | 不影响功能；如需要可考虑 base64 压缩，本期不做 |
| 列宽固定不灵活 | 不同管理员可能希望不同列宽优先级 | 用 fr 弹性单位让浏览器自动分配；不实现"列拖拽" |
